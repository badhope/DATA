# Self Reflection Skill

系统性自我反思与持续改进的完整指南。

## 反思类型体系

### 1. 每日反思

| 时间 | 焦点 | 时长 |
|------|------|------|
| 晨间 | 今日计划、目标 | 5-10min |
| 晚间 | 收获、改进 | 10-15min |

### 2. 周回顾

```markdown
## 周回顾模板

### 本周成就
- 完成了什么？
- 有什么突破？

### 本周挑战
- 遇到了什么问题？
- 有哪些遗憾？

### 经验教训
- 学到了什么？
- 什么方法有效？

### 下周计划

- 需要避免的陷阱
```

## 深度反思框架

### 5 Why 分析法

```markdown
## 问题: 项目延期

### 追问1: 为什么延期？
原因: 代码审查发现大量问题

### 追问2: 为什么有大量问题？
原因: 需求理解有偏差

### 追问3: 为什么需求理解有偏差？
原因: 需求评审时没有充分讨论

### 追问4: 为什么评审不充分？
原因: 评审时间太短，匆忙通过

### 追问5: 为什么时间短？
原因: 排期时没有预留足够的评审时间

---

## 根本原因:
排期时未充分考虑评审所需时间

## 改进行动:
- [ ] 排期时增加20%缓冲时间用于评审
- [ ] 建立需求评审 Checklist
- [ ] 关键需求必须产品确认签字
```

### KPT 复盘法

```markdown
## KPT 复盘

### Keep (保持)
- 团队每日站会很有效
- 代码审查提高了质量
- Pair programming 效果不错

### Problem (问题)
- 文档更新滞后
- 测试覆盖率不足
- 边界情况考虑不够

### Try (尝试)
- 下个项目使用文档模板
- 提高测试覆盖率到 80%
- 添加边界情况检查清单
```

## 经验教训提取

### 教训分类矩阵

| 类别 | 描述 | 示例 |
|------|------|------|
| 技术类 | 代码、架构、工具 | 需要提前考虑扩展性 |
| 流程类 | 工作方式、协作 | 代码审查要及时 |
| 沟通类 | 需求、反馈、期望 | 需要定期同步进度 |
| 决策类 | 选择、权衡、优先级 | 要权衡短期和长期 |

### 教训文档模板

```markdown
## 教训 #2026-001

### 背景
项目: 用户认证系统重构
时间: 2026-03
结果: 延期2周上线

### 教训描述
排期时过于乐观，低估了遗留代码迁移的复杂度

### 影响
- 项目延期
- 团队压力增大
- 质量受到影响

### 根因分析
1. 没有充分评估遗留代码质量
2. 过于依赖开发者的口头估计
3. 没有预留缓冲时间

### 可复用建议
- [ ] 重大重构必须先做技术调研
- [ ] 估算时要有经验丰富的开发者参与
- [ ] 预留20-30%的缓冲时间
- [ ] 建立遗留代码评估Checklist

### 相关教训
- #2025-003 (类似问题)
```

## 改进计划

### PDCA 循环

```
    Plan (计划)
       │
       ▼
    Do (执行)
       │
       ▼
    Check (检查)
       │
       ▼
    Act (处理)
       │
       └──► 返回 Plan
```

### 改进行动模板

```markdown
## 改进行动计划

### 目标
在3个月内将项目估算准确度从60%提升到85%

### 当前状态
- 估算准确度: 60%
- 延期项目比例: 50%

### 目标状态
- 估算准确度: 85%+
- 延期项目比例: <20%

---

## 执行计划

### Week 1-2: 收集数据
- [ ] 整理过去6个月的项目数据
- [ ] 分析延期原因分布
- [ ] 识别估算偏差模式

### Week 3-4: 制定方法
- [ ] 选择适合团队的估算方法 (Planning Poker)
- [ ] 建立估算流程
- [ ] 培训团队成员

### Week 5-8: 试点执行
- [ ] 在2个项目上试点新方法
- [ ] 记录执行情况
- [ ] 收集反馈

### Week 9-12: 优化推广
- [ ] 根据试点反馈优化方法
- [ ] 推广到所有项目
- [ ] 建立估算准确性追踪

### 检查点
- Week 4: 检查方法设计
- Week 8: 检查试点效果
- Week 12: 评估整体效果
```

## 模式识别

### 问题模式分析

```markdown
## 重复问题追踪

### 问题模式: 总是低估复杂度

| 日期 | 项目 | 预估 | 实际 | 偏差 |
|------|------|------|------|------|
| 01-15 | A | 2w | 3w | +50% |
| 02-20 | B | 1w | 2w | +100% |
| 03-10 | C | 3w | 5w | +67% |

### 共同特征
- 需求来自同一产品经理
- 都是遗留系统项目
- 团队成员年轻

### 假设
可能是对新团队成员缺乏历史数据校准

### 行动
- [ ] 建立历史数据库
- [ ] 新人项目增加缓冲
- [ ] 复杂项目增加技术调研阶段
```

### 成功模式分析

```markdown
## 成功因素追踪

### 成功项目特征

| 项目 | 按时 | 质量高 | 团队满意 |
|------|------|--------|----------|
| P1 | ✅ | ✅ | ✅ |
| P2 | ✅ | ✅ | ✅ |
| P3 | ✅ | ✅ | ✅ |

### 共同特征
- 有明确的技术负责人
- 需求变更流程规范
- 团队稳定
- 有充足的测试时间

### 复用建议
- [ ] 每个项目指定技术负责人
- [ ] 严格执行需求变更流程
- [ ] 保证测试时间不压缩
```

## 反思质量评估

### 自检清单

- [ ] 是否客观描述了事实？
- [ ] 是否区分了事实和感受？
- [ ] 是否找到了可控因素？
- [ ] 教训是否具体可操作？
- [ ] 改进行动是否SMART？
- [ ] 是否有跟进计划？
