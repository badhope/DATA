# Context

对于复杂问题，直接给出答案往往不够。逐步思考（Chain of Thought）是一种将复杂问题分解为可管理的步骤，通过明确的推理过程得出答案的方法。本 prompt 的核心目标是：**指导 AI 在回答复杂问题前进行结构化的逐步思考，让推理过程透明可见**。

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `question` | 问题 | `"这个函数有什么性能问题？"` |
| `context` | 上下文 | `{code: "...", usage: "..."}` |
| `required_depth` | 所需思考深度 | `"deep"` |

# Usage Notes

1. **步骤适当**：不要过度分解也不要跳过关键步骤
2. **逻辑透明**：每一步都要有清晰的推理逻辑
3. **假设明确**：用到的假设要明确标注
4. **置信度评估**：给出思考结果的置信度
5. **简洁表达**：逐步思考要简洁，避免冗余

# Example Input

```yaml
question: "这个电商系统应该选择微服务还是单体架构？"
context:
  team_size: "10人"
  complexity: "中等"
  timeline: "3个月"
required_depth: "deep"
```

# Example Output

```yaml
thinking_steps:
  - step: 1
    name: "理解上下文"
    analysis: "团队10人，中等复杂度，3个月时间限制"
    conclusion: "需要平衡开发速度和架构扩展性"
  - step: 2
    name: "评估维度"
    analysis: "从多个维度评估：团队能力、业务复杂度、时间约束"
    conclusion: "列出关键评估维度"
  - step: 3
    name: "方案对比"
    analysis: "对比微服务和单体架构在各维度的表现"
    intermediate_conclusions:
      - "微服务：扩展性好但复杂度高"
      - "单体：简单但扩展性有限"

reasoning_chain: |
  团队小(10人) + 时间紧(3个月) + 中等复杂度
  → 单体架构更合适
  → 但需要考虑未来扩展性

final_answer: |
  建议选择单体架构，原因是：
  1. 团队规模小，微服务的管理开销不成比例
  2. 时间紧张，单体开发更快
  3. 中等复杂度，单体完全能支持

  但建议预留扩展接口：
  - 模块化设计，边界清晰
  - API 设计考虑未来拆分可能
```
