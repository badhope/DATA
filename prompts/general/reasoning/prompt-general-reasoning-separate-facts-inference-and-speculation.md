# Context

知识有不同的确定性程度：事实是确定的，推断有一定依据，猜测则是高度不确定的。混合这些不同类型的知识会误导用户。本 prompt 的核心目标是：**指导 AI 系统性地区分和标注事实、推断和猜测，提高回答的透明度**。

# Prompt Body

## 阶段 1：知识分类

### 1.1 知识类型定义

```markdown
## 知识类型

### 事实 (Fact)
```markdown
**定义**: 可以直接验证为真或为假的陈述
**特征**:
- 有明确的证据支持
- 可以被证实或证伪
- 在可预见的未来不会改变

**示例**:
- "Python 是一种编程语言"
- "水的沸点是100°C"
- "这个文件包含100行代码"

**标记**: 使用 【事实】 标记
```

### 推断 (Inference)
```markdown
**定义**: 基于已知信息和逻辑推理得出的结论
**特征**:
- 基于一定的事实依据
- 通过逻辑推理得出
- 有一定的确定性，但可能出错

**示例**:
- "因为代码中有循环，所以可能有性能问题"
- "用户没有报错，说明功能可能正常"

**标记**: 使用 【推断】 标记，包含推断依据
```

### 猜测 (Speculation)
```markdown
**定义**: 没有充分依据的猜想或假设
**特征**:
- 缺乏明确的证据
- 基于可能性而非确定性
- 需要明确标注为"不确定"

**示例**:
- "可能是网络问题导致的"
- "也许用户更喜欢新界面"

**标记**: 使用 【猜测】 标记，明确说明不确定性
```
```

### 1.2 分类标准

```markdown
## 分类标准

### 判断流程
```markdown
1. 这个陈述可以被直接验证吗？
   → 是 → 【事实】
   → 否 → 继续判断

2. 这个陈述是从已知信息逻辑推导出来的吗？
   → 是 → 【推断】 + 标注依据
   → 否 → 继续判断

3. 这个陈述是基于可能性而非确定性的吗？
   → 是 → 【猜测】 + 标注不确定性程度
```

### 证据程度评估
```markdown
**高证据**: 直接观察、官方文档、普遍接受的事实
**中证据**: 逻辑推断、间接证据、专家共识
**低证据**: 个人经验、有限观察、理论推测
**无证据**: 直觉、猜测、假设
```
```

## 阶段 2：事实提取

### 2.1 事实识别

```markdown
## 事实识别

### 从原文中提取事实
| # | 原文 | 提取的事实 | 验证方式 |
|---|------|------------|----------|
| 1 | [原文] | [事实] | [方式] |

### 事实特征
```markdown
**可验证的特征**:
- 具体的数据或数字
- 明确的事件或行为
- 直接的引用或声明
- 可观察的状态

**验证优先级**:
1. 直接验证（文档、代码）
2. 交叉验证（多个来源）
3. 逻辑验证（内部一致性）
```
```

### 2.2 事实标注

```markdown
## 事实标注

### 标注格式
```markdown
【事实】[陈述]

**来源**: [来源]
**验证方式**: [方式]
**置信度**: 100%（如果是事实）
```

### 示例
```markdown
【事实】这段代码的时间复杂度是 O(n²)

**来源**: 代码分析
**验证方式**: 算法分析
**置信度**: 100%
```
```

### 3.1 推断识别

```markdown

### 从原文中识别推断
| # | 原文 | 推断内容 | 推断依据 | 推断有效性 |
|---|------|----------|----------|------------|
| 1 | [原文] | [推断] | [依据] | [有效/无效] |

### 推断结构
```markdown
**推断格式**:

**推断依据**:
- [依据1]

```
```

```markdown

### 推断置信度
```markdown

**依据强度**:
- 强依据: 多个独立来源支持 [高置信度]
- 中依据: 单个来源或间接证据 [中置信度]
- 弱依据: 个人经验或有限观察 [低置信度]

**潜在反例**:
[如果有反例]

**推断置信度**: [高/中/低]
```
```

### 推断有效性
```markdown
**逻辑有效性**: [✅/❌]
**前提真实性**: [✅/❌/❓]
**结论可靠性**: [高/中/低]

**问题**:
[如果推断无效，说明原因]
```
```

## 阶段 4：猜测处理

### 4.1 猜测识别

```markdown
## 猜测识别

### 猜测特征词
```markdown
**可能表示猜测的词**:
- "可能" / "maybe" / "probably"
- "也许" / "perhaps"
- "大概" / "likely"
- "估计" / "猜测" / "guess"
- "看起来" / "seems"
- "似乎" / "appears"
```

### 猜测清单
| # | 猜测内容 | 依据 | 确定性程度 |

```

### 4.2 猜测标注

```markdown
## 猜测标注

### 标注格式
```markdown
【猜测】[猜测内容]

**确定性程度**: 低
**依据**: [简要说明为什么是猜测]
**需要验证**: [如何验证]
**如果正确/错误的影响**: [影响]

### 猜测与推断的区分
```markdown
**【推断】**: 基于现有证据的逻辑结论
"因为X，所以Y" - 有一定的确定性

**【猜测】**: 基于可能性的假设
"可能是Y" - 缺乏充分证据
```

# Usage Notes

1. **严格分类**：事实、推断、猜测必须严格区分
2. **明确标注**：每种类型都要用明确的标记
3. **标注依据**：推断和猜测都要标注依据或不确定性
4. **分层表达**：按可靠性分层组织回答
5. **透明度优先**：宁可多标注，不可少标注

# Example Input

```yaml
statement: "用户反映系统慢，从日志看数据库查询花了3秒，应该是SQL没有优化"
sources:
  - type: "user_feedback"
    content: "系统响应慢"
  - type: "log"
    content: "Query time: 3000ms"
reasoning_chain:
  - type: "observation"
    content: "日志显示查询3秒"
  - type: "inference"
    content: "可能是SQL问题"
```

# Example Output

```yaml
facts:
  - content: "用户反映系统响应慢"
    source: "用户反馈"
  - content: "日志显示数据库查询耗时3000ms"
    source: "系统日志"
  - content: "查询时间超过2秒定义为准慢"
    source: "行业标准"

inferences:
  - content: "数据库查询是系统慢的主要原因"
    basis: "日志显示查询耗时3秒，占响应时间的主要部分"
    confidence: "high"
  - content: "可能存在SQL性能问题"
    basis: "查询时间长通常是SQL未优化的表现"
    confidence: "medium"

speculations:
  - content: "SQL没有优化"
    basis: "没有实际分析SQL执行计划"
    certainty: "low"
    verification: "需要分析SQL执行计划"

knowledge_classification:
  factual_ratio: "30%"
  inference_ratio: "50%"
  speculation_ratio: "20%"

transparency_report: |
  **结论**: 系统慢的主要原因是数据库查询性能问题

  **【事实】**:
  - 用户反馈系统响应慢
  - 日志显示查询耗时3000ms

  **【推断】**:
  - 数据库查询是主要原因 (置信度: 高)
  - 可能存在SQL优化空间 (置信度: 中)

  **【猜测】**:
  - 具体是哪些SQL需要优化 (确定性: 低，需要分析)

  **置信度**: 中等（部分基于推断）
  **建议**: 需要进一步分析SQL执行计划来确定具体优化点
```
