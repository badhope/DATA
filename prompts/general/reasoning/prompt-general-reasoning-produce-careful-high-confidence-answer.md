# Context

对于高风险或重要的回答，需要特别谨慎。高置信度不是来自表达上的自信，而是来自充分的验证和证据支持。本 prompt 的核心目标是：**指导 AI 在给出重要回答前进行充分验证，产生真正的高置信度答案**。

# Prompt Body

## 阶段 1：风险评估

### 1.1 回答风险识别

```markdown
## 回答风险识别

### 风险维度
| # | 维度 | 高风险特征 | 风险等级 |
|---|------|------------|----------|
| 1 | 影响程度 | 影响用户决策、安全、合规 | [高/中/低] |
| 2 | 可纠正性 | 错误难以纠正或代价高昂 | [高/中/低] |
| 3 | 确定性 | 涉及不确定的技术问题 | [高/中/低] |
| 4 | 可验证性 | 难以通过外部验证 | [高/中/低] |

### 风险等级定义
```markdown
**高风险**: 影响决策、安全、合规的错误回答
**中风险**: 可能导致次优决策的错误回答
**低风险**: 影响较小，容易纠正的错误回答
```
```

```markdown
## 回答类型评估

### 回答类型
| # | 类型 | 风险等级 | 验证要求 |
|---|------|----------|----------|
| 1 | 事实性陈述 | [等级] | [要求] |
| 2 | 技术建议 | [等级] | [要求] |
| 3 | 代码/实现 | [等级] | [要求] |
| 4 | 预测/估计 | [等级] | [要求] |
| 5 | 决策建议 | [等级] | [要求] |

### 高风险场景
```markdown
**高风险场景清单**:
1. 提供安全或合规建议
2. 影响技术架构决策
3. 涉及具体数字或日期承诺
4. 提供代码实现建议
5. 用户明确要求准确
```

## 阶段 2：验证策略

### 2.1 验证层次

```markdown
## 验证层次

### 基础验证
```markdown
**适用**: 低风险回答
**要求**:
- 逻辑一致性检查
- 基本事实核对
- 内部一致性检查

**时间投入**: 快速
```

### 中级验证
```markdown
**适用**: 中风险回答
**要求**:
- 基础验证 +
- 多来源交叉验证
- 假设明确和标注
- 边界情况检查

**时间投入**: 中等
```

### 深度验证
```markdown
**适用**: 高风险回答
**要求**:
- 中级验证 +
- 外部来源验证
- 专家知识核对
- 敏感性分析
- 备选方案考虑
- 潜在问题预判

**时间投入**: 充分
```
```

### 2.2 验证方法

```markdown
## 验证方法

### 逻辑验证
```markdown
**方法**: 检验推理链的有效性
**检查**:
1. 前提是否为真？
2. 推理是否有效？
3. 结论是否从前提必然推出？
4. 有无逻辑跳跃？
```

### 事实验证
```markdown
**方法**: 核对事实的准确性
**检查**:
1. 引用是否准确？
2. 数据是否正确？
3. 来源是否可靠？
4. 是否有遗漏？
```

### 交叉验证
```markdown
**方法**: 使用多个独立来源验证
**检查**:
1. 多来源是否一致？
2. 不一致时如何处理？
3. 哪个来源更可靠？
```

### 边界验证
```markdown
**方法**: 测试边界情况
**检查**:
1. 在边界条件下是否成立？
2. 有无反例？
3. 适用范围是什么？
```
```

### 4.1 置信度评估框架

```markdown
## 置信度评估框架

### 置信度等级
| 等级 | 定义 | 证据要求 | 表达方式 |
|------|------|----------|----------|
| 高 | 非常有把握 | 多个强证据支持 | 明确表达 |
| 中 | 有一定把握 | 有证据支持但有限 | 谨慎表达 |

### 置信度评估因素
```markdown

- 来源单一
- 逻辑链不完整
- 与已知事实矛盾

# Usage Notes

1. **验证充分**：高置信度来自充分验证，而非表达上的自信
2. **证据为王**：没有证据支持的声明不能是高置信度
3. **透明不确定性**：明确标注不确定性比隐藏更可靠
4. **区分置信度和确定性**：置信度是对当前状态的评估
5. **诚实评估**：不夸大置信度，诚实面对不确定性

# Example Input

```yaml
question: "这个API调用是否存在线程安全问题？"
context:
  code: "class Counter { private int count = 0; public void increment() { count++; } }"
  usage: "在多线程环境下使用"
risk_level: "high"
```

# Example Output

```yaml
verification_results:
  factual_check:
    status: "pass"
    notes: "代码分析正确识别了非原子操作"
  logical_check:
    status: "pass"
    notes: "race condition逻辑正确"
  boundary_check:
    status: "pass"
    notes: "边界情况考虑充分"

confidence_level: "high"

answer: |
  **结论**: 存在线程安全问题

  **问题分析**:
  - `count++` 操作不是原子的（读取-修改-写入）
  - 多线程环境下会导致数据竞争

  **证据**:
  1. `count` 字段没有 volatile 或 atomic 修饰
  2. `increment()` 方法没有同步
  3. 非原子操作在多线程下不是线程安全的

  **风险**:
  - 可能导致计数不准确
  - 高并发下问题更明显

  **建议**:
  1. 使用 `AtomicInteger` 替代 `int`
  2. 或使用 `synchronized` 同步
  3. 或使用 `ReentrantLock`

confidence_statement: |
  【置信度】: 高

  【依据】:
  - Java内存模型和线程安全是成熟理论
  - `count++` 的非原子性是确定事实
  - 无需外部验证，基于语言规范

  【假设】: 无

  【不确定性】: 无

  【适用】: 标准Java多线程环境
```
