# Context

任何推理都包含假设和不确定性。透明的做法是将这些假设和不确定性显式化，让用户了解结论的可靠程度。本 prompt 的核心目标是：**指导 AI 系统性地识别和标注推理过程中的假设和不确定性，提高回答的透明度**。

# Prompt Body

## 阶段 1：假设识别

### 1.1 假设类型

```markdown
## 假设类型

### 事实性假设
```markdown
**定义**: 关于现实的假设，可以被验证但尚未验证
**示例**: "用户使用的是现代浏览器"、"服务器有足够的内存"
**特征**: 可能是真的也可能是假的
**处理**: 明确标注，让用户确认或验证
```

### 推断性假设
```markdown
**定义**: 从已知信息推断出的假设
**示例**: "如果系统变慢，用户会不满"、"增加功能会提高用户满意度"
**特征**: 基于逻辑或经验的推断
**处理**: 标注推断依据和置信度
```

### 情境性假设
```markdown
**定义**: 关于当前情境的假设
**示例**: "用户需要快速交付"、"团队有相关经验"
**特征**: 依赖于特定情境
**处理**: 明确假设的情境条件
```

### 价值观假设
```markdown
**定义**: 关于优先级和偏好的假设
**示例**: "性能比功能更重要"、"简单比复杂更好"
**特征**: 不是客观事实，而是价值判断
**处理**: 标注并说明这是价值选择
```
```

### 1.2 假设识别方法

```markdown
## 假设识别方法

### 关键词识别
```markdown
**可能包含假设的关键词**:
- "假设" / "assuming"
- "大概" / "probably"
- "可能" / "might"
- "通常" / "usually"
- "应该" / "should"
- "预期" / "expect"
- "估计" / "estimate"
```

### 逻辑识别
```markdown
**推理中的假设位置**:
1. 前提中包含的假设
2. 从前提到结论的推理中包含的假设
3. 结论中隐含的假设
```

### 来源分类
```markdown
| # | 假设来源 | 说明 | 识别难度 |
|---|----------|------|----------|
| 1 | 用户明确声明 | 来自对话历史 | 易 |
| 2 | 上下文隐含 | 从上下文化推 | 中 |
| 3 | 常识性假设 | 不言自明的假设 | 难 |
| 4 | 专业假设 | 需要领域知识 | 中 |
```
```

### 置信度定义
```markdown
**高置信度**: 有明确的证据或普遍接受的事实支持
**中置信度**: 基于推断或部分证据，但有一定不确定性
**低置信度**: 基于猜测或非常有限的证据
```

### 假设风险
```markdown
**高风险假设**: 如果为假，会影响核心结论
**中风险假设**: 如果为假，会影响部分结论
**低风险假设**: 如果为假，影响较小
```
```

## 阶段 3：不确定性分类

### 3.1 不确定性类型

```markdown
## 不确定性类型

### 认知不确定性
```markdown
**定义**: 由于信息不足而产生的不确定性
**示例**: "不知道用户具体的使用场景"
**特征**: 更多信息可以减少
**处理方式**: 收集更多信息或明确标注信息不足
```

### 随机不确定性
```markdown
**定义**: 由于随机性而产生的不确定性
**示例**: "用户行为有一定的随机性"
**特征**: 不可减少，但可以量化
**处理方式**: 概率表达或范围表达
```

### 模型不确定性
```markdown
**定义**: 由于模型或方法局限而产生的不确定性
**示例**: "简化模型可能遗漏重要因素"
**特征**: 更复杂的模型可能改变结论
**处理方式**: 敏感性分析
```

### 3.2 不确定性量化

```markdown
## 不确定性量化

### 量化方法
| # | 类型 | 量化方式 | 示例 |
|---|------|----------|------|
| 1 | 数值范围 | 给出估计范围 | "估计 100-150 万" |
| 2 | 概率 | 给出概率 | "70% 可能性" |
| 3 | 置信区间 | 区间估计 | "95% CI: [100, 150]" |
| 4 | 定性描述 | 高/中/低 | "高不确定性" |

### 不确定性报告格式
```markdown
**估计值**: [值]
**不确定性**: [量化描述]
**依据**: [依据]
**主要不确定来源**: [来源]
```
```

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `statement` | 声明/回答 | `"系统性能会提升"` |
| `reasoning_context` | 推理上下文 | `{facts: [...], inferences: [...]}` |

# Usage Notes

1. **主动识别**：不要等用户问，主动识别和标注假设
2. **区分类型**：区分事实、假设、推断、不确定性
3. **量化置信度**：给出明确的置信度评估
4. **影响说明**：说明假设变化对结论的影响
5. **实用建议**：基于当前置信度给出实用建议

# Example Input

```yaml
statement: "我们应该选择微服务架构，因为更容易扩展"
reasoning_context:
  facts:
    - "用户量增长了5倍"
    - "团队有10个人"
  inferences:
    - "未来用户量还会继续增长"
    - "微服务更容易扩展"
```

# Example Output

```yaml
explicit_assumptions:
  - assumption: "未来用户量会继续增长"
    type: "推断性假设"
    confidence: "medium"
    basis: "基于当前增长趋势"
  - assumption: "微服务比单体更容易扩展"
    type: "专业假设"
    confidence: "high"
    basis: "行业经验"

implicit_assumptions:
  - assumption: "扩展性是当前最重要的问题"
    type: "价值观假设"
    location: "问题定义"
  - assumption: "团队有能力维护微服务架构"
    type: "情境性假设"
    location: "团队能力"

uncertainties:
  - uncertainty: "未来用户增长的具体规模"
    type: "随机不确定性"
    quantification: "未知，但增长趋势明显"
  - uncertainty: "微服务实施的成本是否值得"
    type: "认知不确定性"
    quantification: "需要更详细的成本分析"

confidence_level: "medium"

transparency_report: |
  **结论**: 建议选择微服务架构

  **确定事实**:
  - 用户量增长了5倍

  **假设**:
  - 微服务更容易扩展 (高置信度)
  - 未来用户量会继续增长 (中置信度)
  - 扩展性是当前最重要的问题 (需要确认)

  **置信度**: 中

  **建议**: 在做最终决策前，建议：
  1. 确认扩展性是否是当前最重要的问题
  2. 评估团队是否具备微服务实施能力
  3. 进行更详细的成本效益分析
```
