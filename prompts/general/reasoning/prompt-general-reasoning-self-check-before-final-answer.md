# Context

在给出最终答案前进行系统性自检，可以发现和修正潜在的问题，提高回答的质量。本 prompt 的核心目标是：**指导 AI 在最终答案前进行结构化的自我检查，发现并修正问题**。

# Prompt Body

## 阶段 1：自检准备

### 1.1 自检清单类型

```markdown
## 自检清单类型

### 事实性检查
```markdown
**检查项**:
1. 事实是否准确？
2. 引用是否有依据？
3. 数据是否正确？
4. 来源是否可靠？

**检查方法**:
- 逐一核对事实
- 追溯引用来源
- 交叉验证数据
```

### 逻辑性检查
```markdown
**检查项**:
1. 推理是否有效？
2. 结论是否从前提必然推出？
3. 有无逻辑跳跃？
4. 有无循环论证？

**检查方法**:
- 检验推理链
- 识别假设
- 验证推论有效性
```

### 完整性检查
```markdown
**检查项**:
1. 是否完整回答了问题？
2. 是否遗漏了重要方面？
3. 是否考虑了边界情况？
4. 是否有信息缺口？

**检查方法**:
- 对照问题检查
- 考虑反面意见
- 测试边界情况
```

### 一致性检查
```markdown
**检查项**:
1. 与已知事实是否一致？
2. 与之前的陈述是否一致？
3. 内部是否一致？
4. 与领域知识是否一致？

**检查方法**:
- 交叉对比
- 检查承诺
- 内部逻辑验证
```
```

### 1.2 自检策略

```markdown
## 自检策略

### 策略选择
| 场景 | 策略 | 检查深度 |
|------|------|----------|
| 高风险回答 | 全面检查 | 深入 |
| 常规回答 | 标准检查 | 中等 |
| 简单回答 | 快速检查 | 基础 |
| 不确定时 | 深度检查 | 最深入 |

### 高风险场景
```markdown
**高风险回答特征**:
- 涉及用户决策
- 包含具体数字或日期
- 涉及代码或技术实现
- 涉及安全或合规问题
- 用户明确要求准确
```
```

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `draft_answer` | 初稿答案 | `"建议使用微服务"` |
| `original_question` | 原始问题 | `"应该选择什么架构"` |
| `reasoning_context` | 推理上下文 | `{facts: [...], reasoning: "..."}` |

# Usage Notes

1. **不跳过检查**：即使是简单的回答也要进行基础检查
2. **重点检查高风险项**：对于高风险回答进行深度检查
3. **修正要彻底**：发现问题必须修正，不能只标注不处理
4. **记录检查过程**：让用户知道做了哪些检查
5. **明确置信度**：让用户知道答案的可靠程度

# Example Input

```yaml
draft_answer: "应该使用Redis作为缓存，因为它性能好"
original_question: "缓存方案选择什么好？"
reasoning_context:
  facts:
    - "当前系统访问量大"
    - "团队熟悉Redis"
  reasoning: "Redis性能好，所以应该用"
```

# Example Output

```yaml
check_results:
  factual_check:
    status: "pass"
    notes: "性能好是有依据的"
  logical_check:
    status: "pass"
    notes: "逻辑成立"
  completeness_check:
    status: "partial"
    missing: ["成本因素未考虑", "替代方案未提及"]

issues_found:
  - issue: "成本因素未考虑"
    severity: "medium"
    correction: "补充成本分析"
  - issue: "替代方案未提及"
    severity: "low"
    correction: "简要提及替代方案"

corrections_made:
  - original: "应该使用Redis作为缓存"
    corrected: "在成本可控的情况下，Redis是不错的选择"
    reason: "更准确的表达"

verified_answer: |
  推荐Redis作为缓存方案，原因是：
  1. 性能优秀
  2. 团队熟悉

  但需要注意成本因素，建议先评估：
  - Redis的运营成本
  - 是否有更经济的替代方案

  置信度: 中等 (未考虑成本因素)
```
