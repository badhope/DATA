# Context

在回答问题前，AI 需要检查是否有所需的上下文。如果信息不足，应该明确指出缺口，而非基于假设给出可能错误的回答。本 prompt 的核心目标是：**指导 AI 在回答前识别上下文缺口，明确区分可回答和不可回答的部分**。

# Prompt Body

## 阶段 1：上下文需求分析

### 1.1 所需上下文识别

```markdown
## 所需上下文

### 回答问题所需的上下文
| # | 所需信息 | 类型 | 重要程度 | 来源 |
|---|----------|------|----------|------|
| 1 | [信息] | [类型] | [高/中/低] | [来源] |
```

### 上下文类型
```markdown
**事实性信息**: 需要确认的客观事实
**约束性信息**: 限制条件
**偏好性信息**: 用户的倾向
**情境性信息**: 当前状态和背景
```
```

### 1.2 上下文完整性检查

```markdown
## 上下文完整性检查

### 可用性检查
| # | 所需信息 | 可用性 | 置信度 | 说明 |
|---|----------|--------|--------|------|
| 1 | [信息] | [✅/❌/❓] | [置信度] | [说明] |

### 可用性状态
```markdown
**✅ 已知**: 信息在上下文中明确存在
**❌ 缺失**: 信息在上下文中不存在
**❓ 假设**: 信息不存在但可以合理假设
```
```

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `current_question` | 当前问题 | `"这个函数有什么问题？"` |
| `available_context` | 可用上下文 | `{code: "...", history: "..."}` |
| `required_context` | 所需上下文 | `["函数完整代码", "上下文"]` |

# Usage Notes

1. **不假设关键信息**：关键缺口不应假设，而应澄清
2. **区分事实和假设**：明确标注假设部分
3. **主动提出澄清**：不要等用户问，主动提出需要的澄清
4. **量化置信度**：给出明确的置信度评估
5. **给出可回答的部分**：即使有缺口，也给出能确定的部分

# Example Input

```yaml
current_question: "这段代码有什么性能问题？"
available_context:
  code: "function process(items) { return items.filter(x => x > 0).map(x => x * 2); }"
  language: "JavaScript"
required_context:
  - "代码完整上下文"
  - "使用场景"
  - "数据规模"
```
