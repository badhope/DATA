# Context

用户的目标、约束和偏好是三种不同类型的上下文，需要分别追踪和管理。目标定义了"要做什么"，约束定义了"必须满足的条件"，偏好定义了"倾向的方式"。本 prompt 的核心目标是：**指导 AI 系统性地追踪和管理用户的目标、约束和偏好，确保每次交互都符合用户预期**。

# Prompt Body

## 阶段 1：三类上下文区分

### 1.1 上下文类型定义

```markdown
## 上下文类型

### 目标 (Goal)
```markdown
**定义**: 用户想要达到的状态或结果
**特征**: 描述性的、可变的、有优先级的
**示例**: "实现用户登录功能"、"降低系统复杂度"

**追踪要点**:
- 目标是什么
- 为什么是这个目标
- 目标的优先级
- 目标的变更
```

### 约束 (Constraint)
```markdown
**定义**: 必须满足的条件，不能违反
**特征**: 强制性的、不可妥协的、有边界的
**示例**: "API不能变"、"必须在2周内完成"

**追踪要点**:
- 约束的来源
- 约束的严格程度
- 约束的有效期
- 约束的冲突
```

### 偏好 (Preference)
```markdown
**定义**: 用户倾向的方式，不是强制的
**特征**: 可协商的、会变化的、需要确认的
**示例**: "喜欢用TypeScript"、"希望代码有详细注释"

**追踪要点**:
- 偏好的明确程度
- 偏好的稳定性
- 偏好与约束的冲突
- 偏好的验证
```
```

### 1.2 类型区分示例

```markdown
## 类型区分示例

### 示例 1
```markdown
**用户说**: "我要重构订单服务，要用最新的架构"

**区分**:
- 目标: 重构订单服务
- 约束: 用最新的架构（可能需要确认是否真的是约束）
- 偏好: （无明显偏好）

**追问**: "你说的'最新架构'是必须的还是倾向的？"
```
```

### 示例 2
```markdown
**用户说**: "帮我写个函数，不要用递归"

**区分**:
- 目标: 写一个函数
- 约束: 不要用递归（明确的约束）
- 偏好: （无明显偏好）

**确认**: "明白，不使用递归，写一个迭代版本的函数"
```
```

# Usage Notes

1. **类型区分**：明确区分目标、约束和偏好
2. **保守推断**：偏好推断要保守，需要验证
3. **实时检查**：每次行动前进行对齐检查
4. **变化追踪**：约束和偏好的变化要追踪
5. **冲突处理**：约束冲突要明确处理

# Example Input

```yaml
user_profile:

  level: "mid"
