# Context

在长期协作中，上下文会不断累积，最终变得过长而难以处理。上下文压缩是在保持核心信息的前提下精简表达，让协作继续高效进行。本 prompt 的核心目标是：**指导 AI 智能压缩上下文，保留核心信息的同时精简冗余**。

# Prompt Body

### 信息类型分布
| # | 类型 | 占比 | 压缩优先级 |
|---|------|------|------------|
| 1 | 目标/约束 | 10% | 最高 |
| 2 | 决策/结论 | 15% | 最高 |
| 3 | 技术规格 | 20% | 高 |
| 4 | 讨论细节 | 30% | 中 |
| 5 | 重复确认 | 15% | 低 |

2. **确认膨胀**: 重复确认已知信息
3. **探索膨胀**: 尝试后又放弃的方案

## 阶段 3：压缩策略

### 3.1 压缩方法

```markdown
## 压缩方法

### 归纳压缩
```markdown
**方法**: 将多个讨论归纳为一个结论
**示例**:
Before: "讨论了方案A、方案B、方案C各自的优缺点"
After: "讨论后决定采用方案A，依据是[简要理由]"

**适用**: 讨论过程冗长但结论明确
```

### 引用压缩
```markdown
**方法**: 用"如之前讨论"替代重复内容
**示例**:
Before: "首先需要获取用户列表，然后过滤出活跃用户..."
After: "按之前确定的方式获取用户列表（如之前讨论的[方式]）"

**适用**: 之前已有明确定义的内容
```

### 摘要压缩
```markdown
**方法**: 用一句话总结长篇内容
**示例**:
Before: "张三提出了使用Redis的建议，因为性能好...李四认为也可以用Memcached，因为..." After: "经过讨论，倾向于使用Redis（详情见历史）"

**适用**: 长篇讨论但核心内容简单
```

### 删除压缩
```markdown
**方法**: 直接删除无意义的上下文
**示例**:
Before: "好的，我来帮你处理。让我先看看这个需求。"
After: (直接删除)

**适用**: 客套话、无效确认等
```
```

## 阶段 4：压缩执行

### 4.2 压缩比例

```markdown
## 压缩比例

### 典型压缩比例
| 上下文类型 | 压缩前 | 压缩后 | 压缩比 |
|------------|--------|--------|--------|
| 短对话 (<10轮) | 100% | 100% | 1:1 |
| 中对话 (10-30轮) | 100% | 60-80% | 1:0.7 |
| 长对话 (30-50轮) | 100% | 40-60% | 1:0.5 |
| 超长对话 (>50轮) | 100% | 30-50% | 1:0.4 |

### 压缩原则
```markdown
1. 不丢失任何 P0 约束
2. 不丢失任何决策
3. 不丢失任何承诺
4. 保持上下文可理解性
5. 提供历史索引
```

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `full_context` | 完整上下文 | `{conversation: [...], state: {...}}` |
| `compression_target` | 压缩目标 | `"减少到20轮以内"` |
| `preserved_elements` | 必须保留的元素 | `["P0约束", "决策"]` |

# Usage Notes

1. **保留核心**：不丢失任何 P0 约束和决策
2. **可回滚**：提供历史索引，允许查看原始内容
3. **验证完整**：压缩后进行完整性验证
4. **渐进压缩**：不一次性压缩太多
5. **透明压缩**：标注被压缩的内容

# Example Input

```yaml
full_context:
  conversation_rounds: 45
  elements:
    constraints: ["API不能改", "测试覆盖100%", "用TypeScript"]

compression_target: "减少到20轮以内"
preserved_elements:
  - "所有P0约束"
  - "所有决策"
  - "所有承诺"
```
