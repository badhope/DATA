# Context

用户的请求往往包含隐含假设，这些假设如果错误会导致解决方案不适用。本 prompt 的核心目标是：**指导 AI 系统性地识别用户请求中的隐含假设，并评估这些假设的合理性和风险**。

# Prompt Body

## 阶段 1：假设识别

### 1.1 假设类型

```markdown
## 假设类型

### 环境假设
```markdown
**定义**: 关于技术或业务环境的假设
**示例**:
- "系统运行在云上"
- "用户有管理员权限"
- "数据库是PostgreSQL"

**风险**: 如果错误，整个方案可能不适用
```

### 技术假设
```markdown
**定义**: 关于技术选型或限制的假设
**示例**:
- "可以修改API"
- "支持Java 11以上"
- "可以用第三方库"

**风险**: 如果错误，需要更换技术方案
```

### 业务假设
```markdown
**定义**: 关于业务逻辑或规则的假设
**示例**:
- "用户ID是数字"
- "订单状态只有几种"
- "每天凌晨结算"

**风险**: 如果错误，功能逻辑可能完全不对
```

### 上下文假设
```markdown
**定义**: 关于任务上下文的假设
**示例**:
- "这是生产环境"
- "团队熟悉微服务"
- "这是紧急需求"

**风险**: 如果错误，方案可能不符合实际
```
```

### 1.2 假设识别清单

```markdown
## 假设识别清单

### 从请求中推断假设
```markdown
**请求**: [请求]

**推断的假设**:

**环境假设**:
1. [假设] - 来源: [推断依据]

**技术假设**:
1. [假设] - 来源: [推断依据]

**业务假设**:
1. [假设] - 来源: [推断依据]

**上下文假设**:
1. [假设] - 来源: [推断依据]
```

### 假设识别检查点
```markdown
**检查问题**:
1. 用户指定了技术栈吗？ → 没有则可能有环境假设
2. 用户说明了数据特征吗？ → 没有则可能有业务假设
3. 用户明确了范围吗？ → 没有则可能有上下文假设
4. 用户说明了约束吗？ → 没有则可能有默认假设
```
```

## 阶段 2：假设分析

### 2.1 假设来源

```markdown
## 假设来源分析

### 假设清单
| # | 假设 | 类型 | 来源 | 明显程度 |
|---|------|------|------|----------|
| 1 | [假设] | [类型] | [推断/声明] | [明显/隐含] |

### 来源分类
```markdown
**明确声明**: 用户在请求中直接说明
**合理推断**: 从请求上下文合理推断
**可能推断**: 有多种可能的推断
**未知**: 不确定用户是否假设
```
```

```markdown
## 合理性评估

### 评估标准
```markdown
**高合理性**: 符合常见情况、通用知识
**中合理性**: 有可能但不确定
**低合理性**: 可能性较低、需要验证
```

[同上结构]
