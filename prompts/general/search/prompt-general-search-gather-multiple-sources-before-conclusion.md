# Context

单一来源可能导致偏差，重要结论需要基于多个来源进行交叉验证。本 prompt 的核心目标是：**指导 AI 系统性地收集多个来源，进行交叉验证，得出更可靠的结论**。

# Prompt Body

## 阶段 1：来源规划

### 1.1 来源类型

```markdown
## 来源类型

### 来源分类
| # | 类型 | 示例 | 可靠性 |
|---|------|------|--------|
| 1 | 官方来源 | 官方文档、官方博客 | 高 |
| 2 | 权威来源 | 知名技术博客、专家文章 | 中-高 |
| 3 | 社区来源 | 论坛、Stack Overflow | 中 |
| 4 | 用户来源 | 用户反馈、案例分享 | 低-中 |
| 5 | 商业来源 | 供应商文档、营销内容 | 低-中 |

### 来源多样性
```markdown
**视角多样性**: 
- 开发者视角 vs 用户视角
- 支持方视角 vs 反对方视角
- 技术视角 vs 业务视角
- 理论视角 vs 实践视角
```
```

### 1.2 来源收集策略

```markdown
## 来源收集策略

### 最小来源数量
| # | 场景 | 最小数量 | 建议数量 |
|---|------|----------|----------|
| 1 | 重要决策 | 5个 | 8个以上 |
| 2 | 一般问题 | 3个 | 5个 |
| 3 | 简单确认 | 2个 | 3个 |

### 来源清单
| # | 来源 | 类型 | 预期内容 | 优先级 |
|---|------|------|----------|--------|
| 1 | [来源] | [类型] | [内容] | [高/中/低] |
```
```

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `topic` | 主题 | `"React vs Vue 哪个更好"` |
| `conclusion_needed` | 需要得出的结论 | `"技术选型建议"` |
| `existing_knowledge` | 现有知识 | `"对两者有基本了解"` |

# Usage Notes

1. **来源多样性**：收集不同类型、不同视角的来源
2. **可靠性评估**：评估每个来源的可靠性
3. **交叉验证**：关键事实需要多源确认
4. **矛盾处理**：存在分歧时明确分析原因
5. **透明表达**：清楚表达置信度和不确定性

# Example Input

```yaml
topic: "微服务架构的优缺点"
conclusion_needed: "技术选型建议"
existing_knowledge: "了解基本概念"
```

# Example Output

```yaml
source_analysis:
  - source: "Martin Fowler博客"
    type: "权威技术博客"
    reliability: "high"
    key_points: ["分布式系统复杂性", "独立部署优势", "组织协调成本"]
  - source: "AWS白皮书"
    type: "官方文档"
    reliability: "high"
    key_points: ["微服务最佳实践", "运维复杂性"]
  - source: "开发者社区讨论"
    type: "社区来源"
    reliability: "medium"
    key_points: ["实际项目经验", "团队规模影响"]

cross_validation:
  consistent:
    - "微服务增加运维复杂性"
    - "独立部署是主要优势"
  inconsistent:
    - "何时适合微服务" (取决于团队规模和项目复杂度)

consolidated_conclusion: |
  微服务架构适合：
  - 大型团队
  - 需要独立扩展的服务
  - 有成熟DevOps能力的组织

  不适合：
  - 小型团队
  - 简单应用
  - 缺乏运维能力

confidence_level: "medium"
```
