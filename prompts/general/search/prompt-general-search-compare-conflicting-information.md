# Context

信息冲突是常见现象。回避冲突或简单选择一方都是不专业的做法。本 prompt 的核心目标是：**指导 AI 系统性地识别和分析信息冲突，给出有深度的冲突解决方案**。

# Prompt Body

## 阶段 1：冲突识别

### 1.1 冲突类型

```markdown
## 冲突类型

### 事实冲突
```markdown
**定义**: 对同一事实的不同陈述
**示例**: 来源A说"版本是3.0"，来源B说"版本是3.1"
**处理方式**: 核实哪个来源更权威或更新
```

### 观点冲突
```markdown
**定义**: 对同一问题的不同观点
**示例**: 来源A认为"微服务好"，来源B认为"单体好"
**处理方式**: 分析双方论据，理解冲突根源
```

### 方法冲突
```markdown
**定义**: 实现同一目标的不同方法
**示例**: "应该用Redis" vs "应该用Memcached"
**处理方式**: 分析场景适用性
```

### 优先级冲突
```markdown
**定义**: 对不同因素优先级的不同看法
**示例**: "性能更重要" vs "可维护性更重要"
**处理方式**: 明确场景和权衡
```
```

| 1 | [点] | [观点] | [观点] | [类型] |

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `conflicting_sources` | 冲突来源 | `[{source: "A", view: "微服务好"}]` |
| `conflict_topic` | 冲突主题 | `"微服务vs单体架构"` |

# Usage Notes

1. **不回避冲突**：冲突是正常的，不要回避
2. **深入分析**：分析冲突原因，不只是表面对比
3. **场景化结论**：给出不同场景的建议
4. **透明处理**：清楚说明为什么选择某种观点
5. **承认不确定性**：无法解决时承认不确定性

# Example Input

```yaml
conflicting_sources:
  - source: "技术博客A"
    view: "React性能优于Vue"
  - source: "技术博客B"
    view: "Vue性能优于React"
conflict_topic: "React vs Vue 性能比较"
```

# Example Output

```yaml
conflict_analysis:
  type: "事实冲突+观点冲突"
  severity: "medium"
  points:
    - point: "性能比较"
      source_a_view: "React性能更好"
      source_b_view: "Vue性能更好"
      reason: "测试场景不同、版本不同、测试方法不同"

source_evaluation:
  source_a:
    reliability: "medium"
    issue: "未说明测试环境和版本"
  source_b:
    reliability: "medium"
    issue: "未说明测试环境和版本"

resolution: |
  两者性能差异不大，具体取决于：
  1. 测试场景和条件
  2. 使用的版本
  3. 具体的使用方式

balanced_view: |
  **结论**:
  - React和Vue都是高性能框架
  - 性能差异不是选择的主要因素
  - 应根据团队熟悉度、生态系统选择

  **具体建议**:
  - 小型项目：两者皆可
  - 大型项目：考虑团队经验和生态系统
  - 有特定性能要求：进行实际测试
```
