# Context

对于事实性问题，直接使用训练数据中的知识可能是不准确的或有时效性的。在回答前进行搜索验证可以确保答案的准确性和时效性。本 prompt 的核心目标是：**指导 AI 在回答事实性问题前判断是否需要搜索，并进行必要的搜索验证**。

# Prompt Body

## 阶段 1：搜索必要性判断

### 1.1 问题类型分析

```markdown
## 问题类型分析

### 需要搜索的问题类型
| # | 类型 | 特征 | 搜索必要性 |
|---|------|------|------------|
| 1 | 最新信息 | 涉及时间敏感信息 | 高 |
| 2 | 具体事实 | 需要特定的事实数据 | 高 |
| 3 | 技术细节 | API版本、库版本等 | 高 |
| 4 | 产品/服务 | 特定产品的当前状态 | 高 |
| 5 | 数值数据 | 统计数据、指标等 | 高 |

### 不一定需要搜索的问题类型
| # | 类型 | 特征 | 搜索必要性 |
|---|------|------|------------|
| 1 | 通用概念 | 经典理论、基础原理 | 低 |
| 2 | 逻辑推理 | 可以通过推理解决 | 低 |
| 3 | 主观意见 | 需要个人判断 | 低 |
| 4 | 代码解释 | 基于代码本身 | 低 |
```

### 1.2 知识时效性评估

```markdown
## 知识时效性评估

### 时效性检查
| # | 问题领域 | 知识更新时间 | 时效性风险 |
|---|----------|--------------|------------|
| 1 | 编程语言版本 | 持续更新 | 高 |
| 2 | 框架/库版本 | 频繁发布 | 高 |
| 3 | 云服务功能 | 经常变化 | 高 |
| 4 | 算法理论 | 相对稳定 | 低 |
| 5 | 设计模式 | 相对稳定 | 低 |

### 时效性判断
```markdown
**问题**: [问题]

**知识时效性评估**:
- 最后更新时间: [时间]
- 更新频率: [频率]
- 时效性风险: [高/中/低]

**是否需要搜索**: [是/否]
**理由**: [理由]
```

```markdown

**影响**: 无法提供准确答案

**处理方式**:

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `question` | 问题 | `"Python 3.11有什么新特性？"` |
| `question_type` | 问题类型 | `"最新信息"` |
| `current_knowledge` | 当前知识 | `"关于Python 3.10的知识"` |

# Usage Notes

1. **先判断再搜索**：不是所有问题都需要搜索
2. **明确搜索目的**：带着具体问题搜索
3. **多源验证**：重要信息需要多源验证
4. **透明来源**：明确标注信息来源
5. **处理不确定性**：搜索失败或冲突时的处理

# Example Input

```yaml
question: "Python 3.12相比3.11有什么新特性？"
question_type: "技术版本比较"
current_knowledge: "了解Python 3.11特性"
```
