name: Refine Ambiguous Request
summary: 精炼歧义请求，将模糊、不清晰的用户请求转化为明确的行动指令

intent: |
  用于识别和消除用户请求中的歧义，将模糊描述转化为明确、可执行的具体指令。
  重点是主动识别歧义点，并通过提问或推断来澄清。
applicable_models:
  - "*"
required_inputs:
  - ambiguous_request: string 存在歧义的用户请求
  - context: string 可用的上下文信息（可选）
outputs:
  - refined_request: string 精炼后的明确请求
  - identified_ambiguities: array 识别出的歧义点
  - clarification_questions: array 澄清问题
  - assumptions: array 做出的假设
steps:
  - id: 1
    name: 识别歧义类型
    action: |
      逐句分析请求，识别歧义类型：

      **类型 1：词汇歧义**
      - 多义词导致的不同理解
      - 专业术语的模糊使用
      - 例："处理数据" - 什么处理？清洗/分析/转换？

      **类型 2：范围歧义**
      - 不明确包含什么、不包含什么
      - 例："优化这个函数" - 只优化这一个还是连带相关？

      **类型 3：标准歧义**
      - 成功标准不明确
      - 例："写得好看" - 什么标准叫好看？

      **类型 4：上下文歧义**
      - 缺少必要的上下文
      - 例："像之前那样做" - 哪个之前？

      对每个歧义点：
      - 标注位置
      - 说明可能的理解
      - 列出需要的澄清
    output: 歧义清单

  - id: 2
    name: 评估信息完整性
    action: |
      检查是否有足够信息来执行：

      **必须明确的要素**：
      - ✅ 目标：用户想达成什么
      - ✅ 范围：要处理什么
      - ✅ 标准：如何判断成功
      - ✅ 约束：有什么限制

      **可能缺失的要素**：
      - 输入数据的格式
      - 输出格式要求
      - 技术栈/环境限制
      - 时间/资源限制

      评估结果：
      - 充足：可以推断执行
      - 部分缺失：需要提问或假设
      - 严重缺失：无法执行，需要完整澄清
    output: 完整性评估

  - id: 3
    name: 生成澄清问题
    action: |
      针对每个歧义点，生成具体问题：

      **问题生成策略**：
      - 用封闭式问题确认具体细节
      - 用开放式问题探索可能性
      - 提供选项帮助用户选择

      **示例**：
      - "你说的'优化'是指：[A] 提升性能 / [B] 提升可读性 / [C] 两者都要？"
      - "这个数据是来自：[A] 数据库 / [B] API / [C] 本地文件？"
      - "输出格式需要是：[A] JSON / [B] CSV / [C] 其他？"

      按优先级排序问题：
      1. 阻塞性问题（必须回答才能继续）
      2. 重要问题（影响质量）
      3. 确认性问题（可选，帮助更好执行）
    output: 澄清问题清单

  - id: 4
    name: 做出合理假设
    action: |
      在请求澄清的同时，做出合理假设：

      **假设原则**：
      1. 基于常见场景假设
      2. 基于上下文假设
      3. 标注假设内容
      4. 说明假设依据

      **假设格式**：
      ```markdown
      假设：[假设内容]
      依据：[为什么这样假设]
      风险：[如果假设错误的影晌]
      验证：[如何验证这个假设]
      ```

      **常见默认假设**：
      - 语言：使用项目主要语言
      - 格式：优先结构化格式
      - 风格：遵循项目现有风格
      - 范围：最小可用范围
    output: 假设清单

  - id: 5
    name: 生成精炼版本
    action: |
      综合澄清和假设，生成精炼版本：

      ```markdown
      # 精炼后的请求

      ## 目标
      [明确的目标描述]

      ## 范围
      - 包括：[明确包括的内容]
      - 不包括：[明确排除的内容]

      ## 标准
      - 成功标准：[如何判断完成]
      - 质量标准：[什么算质量好]

      ## 约束
      - [约束1]
      - [约束2]

      ## 假设
      - [假设1]
      - [假设2]

      ## 待确认
      - [问题1]
      - [问题2]
      ```
    output: 精炼版本

  - id: 6
    name: 输出最终结果
    action: |
      最终输出格式：

      ```markdown
      ## 精炼后的请求
      [完整描述]

      ## 识别的歧义及处理
      | 歧义点 | 可能理解 | 处理方式 |
      |--------|----------|----------|
      | ... | ... | ... |

      ## 待澄清问题
      1. [问题1]
      2. [问题2]

      ## 基于假设的内容
      - [假设内容]（需要确认）
      ```
    output: 最终结果

anti_patterns: |
  - 不要假设用户知道歧义在哪里，要明确指出来
  - 不要跳过歧义直接执行，可能做错
  - 不要问太多问题，分批次问，先问最关键的
  - 不要不做假设就等待，要主动推断并标注
  - 不要假设所有用户都喜欢同样的方式，要根据上下文判断

failure_modes: |
  - **歧义识别不全**：有遗漏 → 回顾常见歧义类型，检查是否都覆盖
  - **假设错误**：假设导致结果不符预期 → 明确标注假设，请求确认
  - **问题过于技术化**：用户无法理解 → 用通俗语言解释

self_check: |
  - [ ] 是否识别了所有明显的歧义？
  - [ ] 澄清问题是否具体可回答？
  - [ ] 假设是否有合理依据？
  - [ ] 假设是否被明确标注？
  - [ ] 精炼版本是否明确可执行？
  - [ ] 是否区分了"必须澄清"和"最好澄清"？
