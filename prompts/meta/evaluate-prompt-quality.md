intent: |
  用于系统性评估 Prompt 的质量，发现问题并给出改进建议。
  涵盖完整性、清晰度、可执行性、可维护性等多个维度。
applicable_models:
  - "*"
required_inputs:
  - prompt_to_evaluate: string 要评估的 Prompt
  - evaluation_focus: string 评估重点（可选）
outputs:
  - quality_score: object 各维度评分
  - issues_found: array 发现的问题
  - recommendations: array 改进建议
  - overall_verdict: string 总体评价
steps:
  - id: 1
    name: 结构完整性检查
    action: |
      检查 Prompt 是否包含所有必要要素：

      **标准 Prompt 结构**：
      - [ ] 角色定义（AI 应该是什么身份）
      - [ ] 任务描述（具体要做什么）
      - [ ] 输入说明（用户提供什么）
      - [ ] 输出要求（期望什么输出）
      - [ ] 约束条件（有什么限制）
      - [ ] 上下文（如需要额外背景）

      **加分项**：
      - [ ] 自检清单（self_check）
      - [ ] 失败模式说明（failure_modes）
      - [ ] 示例（input/output examples）
      - [ ] 相关 Prompts 引用

      **评分**：
      - 完整（80-100%）：核心要素全部包含
      - 基本完整（60-79%）：包含大部分核心要素
      - 不完整（<60%）：缺失重要要素

      输出：
      ```markdown
      完整性得分：X%
      缺失要素：[列表]
      建议：[如何补全]
      ```
    output: 完整性评估

  - id: 2
    name: 清晰度检查
    action: |
      检查描述是否清晰无歧义：

      **歧义检查**：
      - [ ] 术语是否明确？有无多义词？
      - [ ] 指令是否具体？可执行程度？
      - [ ] 约束是否清晰？无模糊表述？
      - [ ] 格式要求是否明确？

      **歧义模式**：
      - ❌ "写得好看" - 什么标准？
      - ✅ "代码格式符合 PEP8 规范"

      - ❌ "处理数据" - 怎么处理？
      - ✅ "对数据进行清洗：去除空值、统一格式"

      **检查方法**：
      - 逐句阅读，看是否能明确执行
      - 标记所有模糊表述
      - 替换为明确表述

      评分：
      - 清晰（80-100%）：几乎无歧义
      - 基本清晰（60-79%）：有少量可改进处
      - 不清晰（<60%）：存在明显歧义
    output: 清晰度评估

  - id: 3
    name: 可执行性检查
    action: |
      检查 Prompt 是否能被 AI 正确执行：

      **执行性检查**：
      - [ ] 任务是否在 AI 能力范围内？
      - [ ] 指令是否足够指导执行？
      - [ ] 是否有矛盾指令（让做又不让做）？
      - [ ] 边界条件是否明确？

      **常见执行问题**：
      - 任务超出 AI 能力（如访问实时数据）
      - 指令模糊导致不同理解
      - 矛盾指令让 AI 难以选择

      **验证方法**：
      - 模拟执行一遍，看是否顺畅
      - 检查是否有"死循环"或"无法结束"
      - 确认输入输出是否匹配

      评分：
      - 可执行（80-100%）：能顺利执行
      - 基本可执行（60-79%）：有少量障碍
      - 难以执行（<60%）：存在执行障碍
    output: 可执行性评估

  - id: 4
    name: 可维护性检查
    action: |
      检查 Prompt 是否易于维护和扩展：

      **维护性检查**：
      - [ ] 结构是否清晰易读？
      - [ ] 是否模块化（各部分职责明确）？
      - [ ] 命名是否规范一致？
      - [ ] 注释是否充分？
      - [ ] 版本是否标注？

      **可维护性特征**：
      - 结构化分区清晰
      - 关键部分有注释
      - 使用标准命名
      - 包含版本信息

      评分：
      - 易维护（80-100%）
      - 基本可维护（60-79%）
      - 难维护（<60%）
    output: 可维护性评估

  - id: 5
    name: 综合评分和报告
    action: |
      汇总各维度评分：

      **评分汇总**：
      | 维度 | 得分 | 权重 |
      |------|------|------|
      | 完整性 | X% | 25% |
      | 清晰度 | X% | 30% |
      | 可执行性 | X% | 30% |
      | 可维护性 | X% | 15% |

      **加权总分**：
      `总分 = 完整性×0.25 + 清晰度×0.3 + 可执行性×0.3 + 可维护性×0.15`

      **等级评定**：
      - A（90-100%）：优秀，直接可用
      - B（75-89%）：良好，轻微改进
      - C（60-74%）：可用，需改进
      - D（45-59%）：勉强可用，明显问题
      - F（<45%）：不合格，需重写

      输出综合报告
    output: 综合评分

  - id: 6
    name: 输出最终结果
    action: |
      最终输出：

      ```markdown
      # Prompt 质量评估报告

      ## 总体评分
      - **等级**：X（A/B/C/D/F）
      - **总分**：XX/100

      ## 各维度评分
      | 维度 | 得分 | 评价 |
      |------|------|------|
      | 完整性 | XX% | [评价] |
      | 清晰度 | XX% | [评价] |
      | 可执行性 | XX% | [评价] |
      | 可维护性 | XX% | [评价] |

      ## 发现的问题
      1. [问题1]
      2. [问题2]

      ## 改进建议
      1. [建议1]
      2. [建议2]

      ## 结论
      [是否推荐使用，需要做什么修改]
      ```
    output: 最终报告

anti_patterns: |
  - 不要只给总分，不给各维度分析
  - 不要回避问题，分数虚高
  - 不要只批评不给出改进建议
  - 不要忽略可执行性（最影响实际使用）

failure_modes: |
  - **评分过于主观**：没有依据 → 使用结构化的检查清单
  - **问题遗漏**：没发现实际问题 → 对照检查清单逐项验证
  - **建议不可行**：改进建议无法执行 → 确保建议具体可操作

self_check: |
  - [ ] 是否覆盖了所有评估维度？
  - [ ] 每个维度是否有具体检查项？
  - [ ] 评分是否有依据？
  - [ ] 建议是否具体可执行？
  - [ ] 报告是否清晰易懂？
