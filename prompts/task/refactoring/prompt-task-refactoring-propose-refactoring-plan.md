# Context

重构是一项需要谨慎规划的工作。没有计划的重构往往会导致范围蔓延、引入 bug 或破坏系统稳定性。本 prompt 的核心目标是：**指导 AI 制定系统性、可验证的重构计划**。

当需要进行任何有意义的重构工作时，都应该先制定计划。

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `target_code` | 目标代码路径 | `src/services/order.service.ts` |
| `refactoring_goals` | 重构目标列表 | `["降低圈复杂度", "消除重复代码"]` |
| `constraints` | 约束条件 | `{time: "2天", scope: "仅内部重构"}` |

# Usage Notes

1. **分析先行**：充分理解代码现状再制定计划
2. **范围明确**：清楚定义包含和排除的范围
3. **风险可见**：识别所有已知风险
4. **验证明确**：制定可执行的验证标准
5. **可回滚**：始终准备回滚方案

# Example Input

```yaml
target_code: "src/utils/format.helper.ts"
refactoring_goals:
  - "降低函数复杂度"
  - "消除重复的格式化逻辑"
  - "提高可测试性"
constraints:
  time: "1天"
  scope: "仅 utils 目录"
```

# Example Output

```yaml
refactoring_plan:
  phases:
    - name: "准备"
      tasks: ["代码分析", "补充测试"]
      duration: "2小时"
    - name: "执行"
      tasks: ["提取函数", "消除重复"]
      duration: "4小时"
    - name: "验证"
      tasks: ["运行测试", "代码审查"]
      duration: "2小时"

scope_definition:
  included:
    - "src/utils/format.helper.ts"
    - "相关测试文件"
  excluded:
    - "调用方代码"
    - "API 接口"

risk_mitigation:
  - risk: "破坏调用方"
    mitigation: "接口保持兼容，通过集成测试验证"
  - risk: "引入新 bug"
    mitigation: "每步验证，测试先行"

verification_criteria:
  - "所有单元测试通过"
  - "测试覆盖率 > 80%"
  - "无 lint 错误"
  - "行为与重构前完全一致"
```
