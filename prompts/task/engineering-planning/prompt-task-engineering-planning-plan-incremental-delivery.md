# Context

增量交付是将大型功能拆分为多个可独立交付的增量，每个增量都能提供价值并可验证。这种方式降低了交付风险，让价值尽早交付。本 prompt 的核心目标是：**指导 AI 规划合理的增量交付策略，确保每个阶段都有价值且可控**。

## 阶段 2：增量划分

### 2.1 增量规划原则

```markdown
## 增量划分原则

### MVP 原则
| 原则 | 说明 | 应用 |
|------|------|------|
| 有价值 | 提供可用的核心价值 | 第一个增量必须可用 |
| 可验证 | 用户反馈验证假设 | 内置反馈机制 |
| 可执行 | 技术上可实现 | 技术可行性评估 |
| 完整 | 端到端功能闭环 | 避免半成品 |

### 增量划分标准
| 标准 | 说明 | 检查问题 |
|------|------|----------|
| 价值独立性 | 每个增量可独立提供价值 | 增量依赖是否太强？ |
| 技术可行性 | 每个增量技术可实现 | 是否存在技术障碍？ |
| 可测试性 | 每个增量可独立测试 | 测试是否需要其他增量？ |

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `feature_description` | 功能描述 | `"电商搜索功能优化"` |
| `full_scope` | 完整范围 | `{core: [...], extended: [...]}` |
| `constraints` | 约束条件 | `{deadline: "2个月", team: "3人"}` |

# Usage Notes

1. **每个增量有价值**：每个增量都能独立提供用户价值
2. **合理的大小**：增量大小应该在 1-2 周内完成
3. **松耦合设计**：阶段之间应该是松耦合的
4. **可回滚**：每个增量都应该可以回滚
5. **持续验证**：每个增量后都要验证

# Example Input

```yaml
feature_description: "用户画像系统"
full_scope:
  core:
    - "用户数据采集"
    - "基础标签生成"
  extended:
    - "智能推荐"
    - "画像分析"
constraints:
  deadline: "3个月"
  team: "4人"
```
