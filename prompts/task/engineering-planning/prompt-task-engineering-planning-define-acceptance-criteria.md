# Context

验收标准是判断功能是否完成的依据。明确的验收标准让开发、测试、产品对"完成"有一致的理解，避免交付后的争议和质量问题了。本 prompt 的核心目标是：**指导 AI 为功能定义清晰、具体、可验证的验收标准**。

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `feature_description` | 功能描述 | `"用户登录功能"` |
| `requirements` | 需求 | `[{id: "REQ-001", description: "支持手机号登录"}]` |
| `quality_goals` | 质量目标 | `{performance: "< 200ms"}` |

# Usage Notes

1. **具体可测量**：避免模糊描述，使用具体数值
2. **可验证**：每条标准都要有明确的验证方法
3. **全面覆盖**：覆盖功能、性能、安全、可用性等
4. **多方确认**：验收标准需要多方达成共识
5. **持续更新**：随着需求变化更新验收标准

# Example Input

```yaml
feature_description: "用户登录功能"
requirements:
  - id: "REQ-001"
    description: "支持手机号验证码登录"
  - id: "REQ-002"
    description: "支持密码登录"
quality_goals:
  performance: "响应时间 < 200ms"
  security: "密码加密存储"
```

# Example Output

```yaml
acceptance_criteria:
  - id: "AC-001"
    description: "手机号验证码登录"
    type: "功能"
    priority: "P0"
    verification: "输入手机号，发送验证码，验证登录"
    pass_conditions:
      - "收到验证码"
      - "验证码正确"
      - "登录成功"
  - id: "AC-002"
    description: "密码登录"
    type: "功能"
    priority: "P0"
    verification: "输入正确密码，验证登录"
    pass_conditions:
      - "密码正确"
      - "登录成功"
      - "返回 Token"

definition_of_done:
  - code: "代码已提交"
    verified_by: "Git"
  - test: "单元测试覆盖率 > 80%"
    verified_by: "测试报告"
  - review: "代码审查通过"
    verified_by: "PR 审批"

verification_methods:
  - method: "功能测试"
    cases: 15
    pass_rate: "100%"
  - method: "性能测试"
    tps: 1000
    latency: "< 200ms"
```
