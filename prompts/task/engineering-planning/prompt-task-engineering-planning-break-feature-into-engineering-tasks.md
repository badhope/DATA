# Context

将功能拆解为具体的工程任务是项目规划和执行的关键步骤。合理的任务拆分让开发工作清晰可控，便于追踪进度、管理依赖、分配资源。本 prompt 的核心目标是：**指导 AI 将功能系统化地拆解为可执行的工程任务**。

## 阶段 4：工作量估算

### 4.1 估算方法

```markdown
## 工作量估算

### 估算维度
| 维度 | 说明 | 权重 |
|------|------|------|
| 基础时间 | 完成任务的理论时间 | 50% |
| 复杂度 | 代码复杂度调整 | 20% |
| 不确定性 | 需求清晰度调整 | 15% |
| 缓冲 | 风险缓冲 | 15% |

### 估算公式
```
实际估算 = 基础时间 × 复杂度系数 × 不确定性系数 × (1 + 缓冲率)
```

### 复杂度系数
| 复杂度 | 系数 | 说明 |
|--------|------|------|
| 低 | 0.8 | 简单 CRUD |
| 中 | 1.0 | 标准业务逻辑 |
| 高 | 1.3 | 复杂算法/新领域 |
| 很高 | 1.5 | 探索性/不确定性高 |
```

# Variables

| 变量 | 说明 | 示例 |
|------|------|------|
| `feature_description` | 功能描述 | `"用户画像分析功能"` |
| `feature_scope` | 功能范围 | `{in: ["采集", "存储"], out: ["分析报告"]}` |
| `team_capacity` | 团队容量 | `{backend: "2人", frontend: "1人"}` |

# Usage Notes

1. **任务大小适中**：每个任务应该是 2-8 小时的工作量
2. **验收标准明确**：每个任务都有明确的验收标准
3. **依赖关系清晰**：任务间的依赖要明确
4. **合理缓冲**：为不确定的任务预留缓冲时间
5. **可追踪**：任务要便于追踪和管理

# Example Input

```yaml
feature_description: "订单历史记录功能"
feature_scope:
  in_scope:
    - "订单列表展示"
    - "订单详情查看"
    - "订单筛选过滤"
  out_of_scope:
    - "订单导出"
team_capacity:
  backend: "2人"
  frontend: "1人"
```
