# Context

测试策略是指导整个项目测试工作的顶层设计。一个好的测试策略应该与项目特点、技术栈、团队能力、业务需求相匹配。本 prompt 的核心目标是：**指导 AI 为项目或功能设计合理的测试策略**。

## 阶段 3：策略建议

### 3.1 单元测试策略

```markdown
## 单元测试策略

### 策略原则
1. **测试行为而非实现**：测试公共接口，不测试私有方法
2. **单一职责**：每个测试只验证一个行为
3. **隔离依赖**：使用 Mock 隔离外部依赖
4. **快速执行**：单元测试应在秒级完成

### 覆盖率目标
| 模块类型 | 语句覆盖 | 分支覆盖 | 函数覆盖 |
|----------|----------|----------|----------|
| 核心业务 | 80% | 70% | 90% |
| 工具函数 | 90% | 80% | 100% |
| 边缘模块 | 60% | 50% | 70% |

### 测试数据策略
- **原则**: 测试数据应该确定性、可重复
- **管理**: 使用测试数据工厂生成
- **清理**: 测试后自动清理数据
```

### 3.2 集成测试策略

```markdown
## 集成测试策略

### 策略原则
1. **测试接口契约**：验证 API 的输入输出
2. **测试数据交互**：验证数据库读写
3. **测试消息传递**：验证消息队列通信
4. **有限外部依赖**：Mock 外部服务

### 覆盖范围
| 集成类型 | 覆盖目标 | 测试数 |
|----------|----------|--------|
| API 集成 | 关键 API 100% | [N] |
| 数据库集成 | CRUD 操作 100% | [N] |
| 消息集成 | 关键消息 80% | [N] |

### 测试环境策略
- **环境**: 使用测试环境或容器化环境
- **数据**: 使用独立的测试数据库
- **隔离**: 测试间数据隔离
```

### 3.3 E2E 测试策略

```markdown
## E2E 测试策略

### 策略原则
1. **聚焦关键路径**：只测试核心用户流程
2. **真实环境**：使用接近生产的测试环境
3. **用户视角**：模拟真实用户操作
4. **稳定性优先**：避免脆弱的 E2E 测试

### 关键用户路径
| # | 路径 | 优先级 | 自动化目标 |
|---|------|--------|------------|
| 1 | [路径] | P0 | 100% |
| 2 | [路径] | P1 | 80% |

### 执行策略
- **频率**: 每日一次，在 CI/CD 流程中
- **并行**: 关键路径可以并行执行
- **报告**: 失败时自动生成报告和截图
```

# Usage Notes

1. **匹配项目**：策略必须与项目特点匹配
2. **平衡质量**：在质量、速度、成本间平衡
3. **可实现**：目标必须是可实现的
4. **分阶段**：大型项目要分阶段实施
5. **持续改进**：策略应该持续优化
