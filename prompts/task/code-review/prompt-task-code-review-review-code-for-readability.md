# Context

代码可读性直接影响开发效率和协作质量。可读性差的代码难以理解、容易产生 bug、增加维护成本。本 prompt 的核心目标是：**指导 AI 系统化地评估和改进代码可读性**。

### 1.2 命名问题类型

```markdown
## 常见命名问题

### 无意义名称
| # | 示例 | 问题 | 建议 |
|---|------|------|------|
| 1 | `data`, `info` | 太模糊 | 使用具体名称 |
| 2 | `temp`, `tmp` | 无上下文 | 说明用途 |
| 3 | `x`, `y`, `z` | 无语义 | 使用有意义名称 |

### 误导性名称
| # | 示例 | 问题 | 建议 |
|---|------|------|------|
| 1 | `isActive` 但检查权限 | 名不副实 | 改为 `hasPermission` |
| 2 | `getData` 返回修改后的数据 | 命名不准确 | 改为 `fetchAndTransform` |

### 缩写不当
| # | 示例 | 问题 | 建议 |
|---|------|------|------|
| 1 | `usr` | 难以理解 | `user` |
| 2 | `mgr` | 含义模糊 | `manager` |
| 3 | `calcTot` | 不一致 | `calculateTotal` |

### 命名风格不一致
| # | 示例 | 问题 | 建议 |
|---|------|------|------|
| 1 | `userId` vs `user_id` | 混用下划线和驼峰 | 统一风格 |
| 2 | `getUser` vs `fetchUser` | 同一项目风格不一致 | 统一使用一种模式 |
```

# Usage Notes

1. **读者视角**：从阅读者的角度评估
2. **上下文考虑**：考虑代码的功能上下文
3. **平衡评价**：既指出问题也肯定好的实践
4. **具体建议**：每个问题都有具体的改进建议
5. **标准一致**：基于项目或行业标准评估
